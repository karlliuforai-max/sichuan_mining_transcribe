# 四川矿业会议转写 Skill 交接文档

## 1. 当前状态

本项目仍处于方案设计和开发准备阶段，尚未正式创建 skill 或编写处理脚本。当前仓库中已有：

- `任务背景.md`：用户原始需求。
- `转写源/`：四份真实会议转写 PDF。
- `开发文档.md`：本轮讨论后形成的开发指导文档。
- `交接文档.md`：本文件，用于恢复上下文和交接给后续 agent。

## 2. 用户真实诉求

用户是矿业企业负责人，经常参加四川乐山及周边地区的矿业会议，希望把录音先用外部工具转写，再用 AI 对转写文本进行纠错和整理。

难点：

- 参会者大量使用四川话、成都话、乐山话。
- 乐山话口音与普通四川话差异较大，通用转写工具识别效果差。
- 内容高度垂直，涉及矿业、锂矿、选矿、采矿、品位、回收率、尾矿、报价、付款、运输、资质等术语。
- 用户最终需要的不是逐字稿，而是可信、可跟进、可复核的会议纪要。

用户明确要求：

- 输入可以是 PDF 或 TXT。
- 从转写源到会议纪要应连续完成。
- 如果处理超过约 1 分钟，应持续输出进度。
- 要兼容普通话、四川话、乐山话混杂场景。
- 要重点优化矿业领域用词。

## 3. 已形成共识的总体方案

用户认可方案二：`Skill + 脚本 + 术语库 + 可选大模型增强`。

但进一步讨论后，方案被扩展为更通用的三层架构：

1. 核心处理层：跨平台 CLI 或 HTTP 服务。
2. 知识库层：术语表、错词表、prompt、会议纪要模板等纯文本资料。
3. Agent 适配层：Codex、Claude Code、OpenClaw、飞书等调用入口。

重要共识：

- 不应绑定 Codex 独有能力。
- OpenClaw 也支持 `/skill` 调用，因此 skill 文件应尽量通用。
- 飞书 + OpenClaw 是用户最高频使用场景。
- macOS/Windows 本地使用 Codex 或 Claude Code 是典型场景。
- 阿里云服务器部署也是未来可能场景。

开工前进一步确认的默认边界：

- 先做跨平台 CLI，不急着做 HTTP 服务。
- DeepSeek/API 不作为默认上传路径，只有用户显式选择 `deepseek` 或 `auto` 并配置 API key 才调用。
- 每次输出同时保留纠错全文和会议纪要。
- 私有知识库默认跟项目目录走，即项目内 `knowledge/`；同时预留 `--knowledge-dir`，以后可以指向共享目录或服务器知识库。
- 飞书文档写入暂缓到飞书部署阶段单独实现；当前只需保证输出 Markdown/CSV/JSONL 结构稳定，后续由飞书 CLI 写入飞书文档和云盘。

## 4. 关于错词映射的共识

错词映射是必要的，但不能作为唯一纠错方式。

已观察到的典型错词：

- `品味/频率 -> 品位`
- `月矿/圆矿/原款 -> 原矿`
- `金粉/尽分 -> 精粉`
- `李辉时/铝灰石 -> 锂辉石`
- `假常识/价常识/驾乘师 -> 钾长石`
- `淘宝/逃宝/套碗 -> 套保`
- 数字后面的 `人` 在矿石上下文中常可能是 `吨`

设计原则：

- 高置信错词可以规则化处理。
- 需要上下文判断的内容交给模型或进入待复核。
- 不确定内容不要强行改到“看起来通顺”。
- 关键数字、合同、付款、资质、人名、公司名、矿山名必须保守。

## 5. 关于自学习的共识

用户希望 skill 能跟随时间和经验进步。已达成共识：

采用“候选学习 -> 审核 -> 晋升”机制，而不是让模型自动直接修改正式错词表。

建议文件：

- `knowledge/confirmed_corrections.jsonl`：正式确认错词表。
- `knowledge/learned_candidates.jsonl`：新发现候选。
- `knowledge/uncertain_items.jsonl`：不确定项。
- `knowledge/session_history.jsonl`：任务历史。

知识库位置解释：

- `references/` 是内置知识库，随 skill 发布，适合放通用术语、默认错词规则、prompt 和模板。
- `knowledge/` 是用户私有知识库，适合放使用过程中积累的人名、客户名、矿山名、错词候选和已确认错词。
- MVP 默认使用项目内 `knowledge/`，便于复制和迁移；以后可用 CLI 参数 `--knowledge-dir` 指向共享目录或云端同步目录。

候选晋升条件：

- 用户确认。
- 多次会议重复出现。
- 上下文稳定。
- 目标词已在术语表中。
- 置信度足够高。
- 不属于人名、公司名、合同金额等高风险字段。

## 6. 关于 DeepSeek/API 的共识

用户提出可提供 DeepSeek API。讨论结论：

- DeepSeek 值得接入，但应作为可选增强后端。
- 不应让 DeepSeek 取代本地规则和错词表。
- 最佳方案是混合式：
  1. 本地规则预清洗。
  2. 大模型上下文纠错。
  3. 本地校验关键数字和单位。
  4. 生成纪要。
  5. 提取候选错词，但不直接晋升正式表。

推荐模式：

- `local`：只用本地规则。
- `agent`：由当前 agent 按 skill 处理。
- `deepseek`：调用 DeepSeek API。
- `auto`：默认，简单片段本地处理，复杂片段调用模型。

API key 应通过环境变量读取，例如：

```text
DEEPSEEK_API_KEY
```

## 7. 关于会议纪要模板的共识

用户询问是否有 `find-skills` skill 以寻找专业会议纪要模板。当前环境没有名为 `find-skills` 的 skill。可用的是 `skill-installer`，它用于列出和安装 Codex skills，不是会议模板检索工具。

已经参考公开会议纪要模板的共性结构，并决定做成矿业定制模板，而不是照搬普通办公模板。

推荐纪要结构：

1. 会议信息。
2. 会议结论。
3. 关键议题摘要。
4. 关键数据与商务条件。
5. 技术与生产要点。
6. 风险与待确认事项。
7. 行动项。
8. 转写纠错疑点。

矿业场景应特别关注：

- 原矿/精粉品位。
- 回收率。
- 选矿工艺。
- 尾矿处理。
- 运输与堆场。
- 产能约束。
- 报价公式。
- 付款方式。
- 采矿资质和合规风险。
- 套保、税费和价格波动。

参考来源：

- Adobe Acrobat meeting minutes templates：
  https://www.adobe.com/acrobat/business/resources/meeting-minutes-templates.html
- Asana meeting minutes template：
  https://asana.com/templates/meeting-minutes
- Project Management Docs meeting minutes：

## 9. 后续开发建议

建议下一位 agent 从以下顺序开始：

1. 阅读 `任务背景.md`、`开发文档.md`、`交接文档.md`。
2. 抽取四份 PDF 的文本，确认当前环境可用的 PDF 解析库。
3. 创建 skill 目录结构。
4. 实现最小管线：
   - PDF/TXT 提取。
   - 文本清洗。
   - 本地错词规则。
   - 会议纪要模板输出。
   - 进度 JSON Lines。
   - 同时输出纠错全文和会议纪要。
5. 用四份样本跑通。
6. 再接入 DeepSeek API。
7. 最后做 OpenClaw/飞书适配。飞书写入飞书文档和云盘的逻辑留到飞书部署阶段，通过飞书 CLI 单独实现。

第一版不要追求完美自动纠错。更重要的是：

- 全流程可跑通。
- 输出结构稳定。
- 有待复核清单。
- 不乱改关键数字。
- 知识库能逐步积累。

## 10. 需要避免的坑

- 不要依赖 macOS 专属工具，如 `textutil`。
- 不要只做 Codex skill，而忽略 OpenClaw 和 Claude Code。
- 不要让模型直接写正式错词表。
- 不要把 API key 写入仓库。
- 不要在没有证据时强行改人名、公司名、金额、付款条款。
- 不要把会议纪要写成逐字稿摘要，用户需要的是业务决策和跟进事项。
- 不要忽略处理进度，飞书/OpenClaw 高频场景需要进度反馈。

## 11. 可交付物定义

MVP 完成时至少应有：

- 一个通用 `SKILL.md`。
- 可跨平台运行的 Python 处理脚本。
- 初版术语表和错词表。
- 初版会议纪要模板。
- 四份样本的处理输出。
- 进度日志。
- 待复核清单。
- 候选错词学习表。
