# 四川矿业会议转写 Skill 沟通过程时间线

整理日期：2026-05-29

本文档按时间顺序记录本项目从需求讨论、方案设计、MVP 开发、DeepSeek 测试到 GitHub 上传的完整沟通过程。

## 1. 明确原始需求

用户最初提供：

- `任务背景.md`
- `转写源/` 下的 4 份真实会议转写 PDF

核心诉求：

- 处理四川话、乐山话、普通话混杂的矿业会议转写。
- 修复通用转写工具对方言和矿业术语的识别错误。
- 生成可用于业务跟进、复核和归档的会议纪要。
- 重点关注锂矿、选矿、采矿、品位、回收率、精粉、尾矿、运输、报价、付款、资质等内容。

项目定位不是普通“方言翻译器”，而是：

> 矿业会议转写纠错 + 结构化会议纪要生成 + 可复核知识积累。

## 2. 讨论总体方案

最初讨论了几种方案，用户比较认可“方案二”：

> Skill + 脚本 + 术语库 + 可选大模型增强

随后用户补充部署要求：

- 不希望绑定某个 agent 或操作系统。
- 未来可能在 macOS、Windows、阿里云服务器上使用。
- 可能被 Codex、Claude Code、OpenClaw 调用。
- 飞书 + OpenClaw 是高频场景。
- OpenClaw 也支持 `/skill` 调用。

因此架构定为三层：

1. 核心处理层：跨平台 CLI。
2. 知识库层：术语表、错词表、prompt、模板。
3. Agent 适配层：Codex、Claude Code、OpenClaw、飞书。

## 3. 讨论知识积累机制

用户希望 skill 能越用越聪明，而不是固定错词表。

最终确定：

- 自动发现新术语、新错词。
- 写入候选学习文件。
- 不自动晋升正式规则。
- 需要用户确认或满足严格条件后再进入正式知识库。

知识积累机制为：

> 候选学习 -> 人工/规则审核 -> 晋升正式规则

这样可以避免模型把人名、公司名、价格、付款条件等敏感内容学错。

## 4. 讨论 DeepSeek 接入

用户提出可以提供 DeepSeek API，希望大模型参与纠错和纪要生成。

讨论结论：

- 本地规则适合高确定性错词。
- 大模型适合上下文纠错和全文语义纪要。
- 两者应协同，而不是互相替代。

初步模式设计为：

- `local`：只用本地规则。
- `agent`：由当前 agent 继续处理。
- `deepseek`：调用 DeepSeek。
- `auto`：有 key 就用 DeepSeek，没有就本地兜底。

经过实际测试后确认：

> DeepSeek 模式的纪要质量明显优于本地模式，因此默认应改为 DeepSeek。

## 5. 讨论会议纪要模板

用户询问是否有 `find-skills`，希望参考专业会议纪要模板。

当前环境没有专门的 `find-skills` skill，但参考了通用会议纪要模板结构，并结合矿业场景定制模板。

最终纪要结构确定为：

1. 会议信息。
2. 会议结论。
3. 关键议题摘要。
4. 关键数据与商务条件。
5. 技术与生产要点。
6. 风险与待确认事项。
7. 行动项。
8. 转写纠错疑点。

矿业场景中特别关注：

- 品位。
- 回收率。
- 原矿/精粉。
- 运输费用。
- 报价。
- 付款条件。
- 采矿资质。
- 选矿能力。
- 合规风险。

## 8. 本地模式测试

先用 4 份 PDF 跑本地模式。

结果：

- 本地模式跑通。
- 可生成纠错全文、会议纪要、关键数据、行动项和疑点清单。

用户发现的问题：

- 会议纪要里的关键数据太散。
- 商务条件按句子抽取，缺乏整体语义。
- 技术要点、风险、行动项偏碎片化。

结论：

> 本地规则适合作兜底，但不适合作为高质量纪要默认方案。

## 10. 调整 DeepSeek flash/pro 策略

用户建议尝试：

- 默认 flash：`deepseek-v4-flash`
- 高要求 pro：`deepseek-v4-pro`

```bash

--deepseek-tier pro
--deepseek-thinking auto/enabled/disabled
```

初始设计里 pro 默认开启 thinking，但后续测试发现该策略不适合当前 JSON 纠错流程。

## 11. 增加实时进度输出

用户提出等待过程中希望实时看到进度，例如：

- 提取完成，耗时多少秒。
- 纠错进行中。
- 已完成多少字 / 全文多少字。

进度机制调整为：

- 默认 `--progress text`：人类可读中文进度。
- `--progress jsonl`：机器可读 JSON Lines。
- `--progress none`：不打印但仍写日志。

每次运行都会保存：

```text

```

## 16. flash-v2 复测

用新 prompt 重新跑 flash：

```text
sichuan-mining-transcribe/outputs/sample-deepseek-flash-0524-v2/
```

结果：

- 总耗时约 167.8 秒。
- 纠错后约 13,519 字。
- 没有再扩写。
- 纪要质量可用。

结论：

> 新 prompt 成功解决旧 flash 把纠错全文扩写到 2.3 万字的问题，flash 适合作为日常默认模式。

## 17. pro thinking enabled 测试失败

测试：

```text
deepseek-v4-pro + thinking enabled
```

结果：

- 很慢。
- 返回空 `corrected_text`。
- 生成的 `03_corrected.md` 只有 83 字。
- `09_semantic_minutes.json` 基本为空。

结论：

> pro + thinking enabled 当前不适合这个 JSON 纠错流程。

因此增加防护：

- 模型返回空 `corrected_text` 时，自动回退本地纠错 chunk。
- 写入疑点。
- 避免整份纪要变空。

## 18. pro no-thinking 测试成功

测试：

```text
deepseek-v4-pro + thinking disabled
```

输出：

```text
sichuan-mining-transcribe/outputs/sample-deepseek-pro-no-thinking-0524-v2/
```

结果：

- 总耗时约 327.8 秒。
- 纠错全文约 13,491 字。
- 纪要约 3,032 字。
- 没有扩写。
- 纪要更细、更完整。

结论：

> pro no-thinking 适合重要会议，但不适合作为日常默认，因为耗时约为 flash 的 2 倍。
