# 四川矿业会议转写纠错与纪要生成 Skill 开发文档

## 1. 项目目标

开发一个面向四川矿业会议场景的转写后处理 skill。它接收由其他工具生成的会议转写文本或 PDF，针对普通话、成都话、乐山话和矿业术语混杂导致的识别错误进行纠错，并自动生成可用于业务跟进的会议纪要。

该项目不是单纯的“方言翻译器”，而是一个“矿业会议转写修复 + 结构化纪要生成”工作流。

核心目标：

- 支持 PDF/TXT 输入。
- 从输入到输出连续完成，不要求用户中途操作。
- 长任务提供进度信息，例如已处理字数、总字数、当前阶段。
- 重点优化四川话、乐山话口音导致的误识别。
- 重点优化矿业领域词汇，包括锂矿、选矿、采矿、回收率、品位、精粉、尾矿、运输、报价、付款等场景。
- 生成正式会议纪要、关键数据表、风险与待确认事项、行动项。
- 支持知识库随使用积累，但避免错误知识污染正式映射表。
- 尽量不绑定某个操作系统或 agent，便于在 macOS、Windows、Linux/阿里云服务器、Codex、Claude Code、OpenClaw、飞书机器人等环境使用。

## 2. 设计原则

### 2.1 普适部署优先

不要把核心能力写死在某个 agent 的 skill 机制里。推荐采用三层结构：

1. 核心处理层：跨平台 CLI 或 HTTP 服务。
2. 知识库层：纯文本配置和参考资料。
3. Agent 适配层：Codex、Claude Code、OpenClaw、飞书等调用入口。

这样即使某个平台不支持 Codex 风格 skill，也可以通过命令行或 HTTP 服务调用同一套能力。

### 2.2 开工前已确认的实现边界

以下决策已在开工前确认，作为 MVP 默认行为：

- 核心调用方式先固定为跨平台 CLI，后续再按需要封装 HTTP 服务或飞书/OpenClaw 调用。
- 默认隐私策略保守：不自动上传会议内容到外部模型。只有用户显式选择 `deepseek` 或 `auto` 模式，并配置 API key 时，才调用外部模型。
- 每次处理都保留两类主要结果：纠错全文和会议纪要。
- 私有知识库默认跟随项目目录保存，即使用项目内 `knowledge/`。同时 CLI 预留 `--knowledge-dir` 参数，后续可切换到共享目录、云盘目录或服务器统一知识库。
- 飞书写入暂不放入 MVP 主流程。MVP 只保证输出结构稳定的 Markdown/CSV/JSONL 文件；后续部署飞书时，再用飞书 CLI 将会议纪要写入飞书文档并保存到飞书云盘指定位置。

### 2.3 本地规则与大模型协同

错词映射适合处理高确定性错误，大模型适合处理上下文纠错和纪要生成。二者不应互相替代。

推荐流程：

1. 本地规则预清洗。
2. 按语义和长度分块。
3. 将原文片段、相关术语、错词表、上下文摘要发送给模型纠错。
4. 本地校验数字、单位、人名、地名和敏感业务条件。
5. 合并纠错文本。
6. 生成会议纪要。
7. 提取新术语和新错词候选。

### 2.4 自学习但不盲目自动写入

知识库应能随时间进步，但不能把模型猜测直接写入正式错词表。推荐采用“候选学习 -> 审核 -> 晋升”的机制：

- 高置信、重复出现、上下文稳定的错词可自动进入候选表。
- 只有用户确认或满足严格规则后，才晋升到正式映射表。
- 每次任务输出“新增候选”和“待复核疑点”，方便人工维护。

## 3. 推荐架构

### 3.1 目录结构

建议后续正式 skill 使用如下结构：

```text
sichuan-mining-transcribe/
├── SKILL.md
├── agents/
│   └── openai.yaml
├── scripts/
│   ├── transcribe_pipeline.py
│   ├── extract_text.py
│   ├── normalize_text.py
│   ├── apply_rules.py
│   ├── llm_correct.py
│   ├── generate_minutes.py
│   └── update_knowledge.py
├── references/
│   ├── workflow.md
│   ├── glossary.yml
│   ├── correction_rules.yml
│   ├── meeting_templates.md
│   ├── prompt_templates.md
│   └── quality_checks.md
├── knowledge/
│   ├── confirmed_corrections.jsonl
│   ├── learned_candidates.jsonl
│   ├── uncertain_items.jsonl
│   └── session_history.jsonl
└── outputs/
    └── .gitkeep
```

说明：

- `SKILL.md`：通用 agent 入口，写清什么时候使用、如何调用脚本、输出什么。
- `scripts/`：跨平台 Python 脚本，尽量只依赖 Python 标准库和少量可安装依赖。
- `references/`：稳定知识和模板，给 agent 或模型读取。
- `knowledge/`：随使用增长的数据，区分正式确认与候选。
- `outputs/`：每次任务输出纠错文本、纪要、日志和候选学习结果。

### 3.2 跨平台技术建议

优先使用 Python：

- 路径处理使用 `pathlib`。
- PDF 解析使用 `pypdf`，如后续需要更高质量版式解析，可选 `pdfplumber`。
- 不依赖 macOS `textutil`、`pdftotext`、`pdfinfo` 等系统工具。
- 进度输出使用标准输出 JSON Lines，便于飞书、OpenClaw、Codex 和 Claude Code 读取。
- 配置文件使用 YAML 或 JSONL。若想减少依赖，正式运行配置可以使用 JSON/JSONL，模板继续使用 Markdown。

进度输出示例：

```json
{"stage":"extracting","message":"正在提取文本","done_chars":0,"total_chars":0}
{"stage":"correcting","message":"正在纠错第 3/12 段","done_chars":18000,"total_chars":76000}
{"stage":"minutes","message":"正在生成会议纪要","done_chars":76000,"total_chars":76000}
```

## 4. 功能需求

### 4.1 输入

必须支持：

- 单个 PDF。
- 单个 TXT。
- 一个目录下的多个 PDF/TXT。

可选支持：

- Markdown 转写文本。
- 后续扩展 DOCX。

输入文本可能包括：

- 说话人标签。
- 时间戳。
- 自动转写错误。
- 中英日混杂误识别。
- 四川话、乐山话、普通话混杂。
- 矿业、商务、运输、合规术语混杂。

### 4.2 输出

每次处理建议输出一个任务目录：

```text
outputs/2026-05-28_项目名/
├── 01_extracted.txt
├── 02_normalized.txt
├── 03_corrected.md
├── 04_minutes.md
├── 05_key_data.csv
├── 06_action_items.csv
├── 07_uncertain_items.md
├── 08_learn_candidates.jsonl
└── run_log.jsonl
```

其中：

- `03_corrected.md`：纠错后的全文，可保留说话人和时间戳。
- `04_minutes.md`：正式会议纪要。
- `05_key_data.csv`：品位、回收率、吨数、报价、付款比例、运输费用等关键数据。
- `06_action_items.csv`：行动项、责任人、截止时间、产出物。
- `07_uncertain_items.md`：待确认的人名、地名、公司名、术语、数字。
- `08_learn_candidates.jsonl`：新错词候选。

### 4.3 CLI 协议

MVP 先固定 CLI 协议，便于 Codex、Claude Code、OpenClaw、飞书机器人或服务器任务统一调用：

```bash
python scripts/transcribe_pipeline.py \
  --input 转写源 \
  --output outputs/run-001 \
  --mode local \
  --progress jsonl \
  --knowledge-dir knowledge \
  --learn-policy candidate
```

建议参数：

- `--input`：输入文件或目录，支持 PDF/TXT。
- `--output`：输出目录。
- `--mode`：处理模式。
- `--progress`：进度输出格式，MVP 固定支持 `jsonl`。
- `--knowledge-dir`：私有知识库目录，默认 `knowledge/`。
- `--learn-policy`：学习策略，建议支持 `off`、`candidate`、`auto-confirmed`，MVP 默认 `candidate`。
- `--template`：纪要模板，后续可支持 `simple`、`business`、`technical`。

### 4.4 处理模式

建议支持四种模式：

- `local`：只使用本地规则和错词表，不调用外部大模型。隐私最好，效果适中。
- `agent`：由当前 agent 根据 skill 指引完成上下文纠错，不额外调用 API。
- `deepseek`：调用 DeepSeek API 进行纠错和纪要生成。
- `auto`：默认模式。简单片段本地处理，复杂片段调用模型。

命令示例：

```bash
python scripts/transcribe_pipeline.py --input 转写源 --mode auto --output outputs/run-001
```

DeepSeek 模式示例：

```bash
python scripts/transcribe_pipeline.py --input 转写源 --mode deepseek --model deepseek-chat --output outputs/run-001
```

API key 不应写入仓库。建议从环境变量读取：

```text
DEEPSEEK_API_KEY
```

## 5. 纠错策略

### 5.1 已观察到的典型错误

从当前四份真实转写源中观察到的高频错误类型：

- `品味`、`频率` 常应为 `品位`。
- `月矿`、`圆矿`、`原款` 常应为 `原矿`。
- `金粉`、`尽分` 常应为 `精粉`。
- `李辉时`、`铝灰石` 常应为 `锂辉石`。
- `假常识`、`价常识`、`驾乘师` 常应为 `钾长石`。
- `淘宝`、`逃宝`、`套碗` 常应为 `套保`。
- `人` 在“每天五百人”“三万多人”等语境下常应为 `吨`。
- `初设`、`安设`、`环评`、`采矿权`、`资质`、`挂靠` 等词需要作为合规和工程上下文保留。

### 5.2 规则分级

纠错规则应分级处理：

- 强规则：几乎确定的错词，直接替换。例如矿业语境下 `李辉时 -> 锂辉石`。
- 条件规则：需要上下文触发。例如数字后面的 `人` 是否改为 `吨`，要结合 `拉、卖、堆、矿、每天、万吨` 判断。
- 模型规则：需要大模型根据上下文推断。例如人名、地名、公司名、整句语义断裂。
- 保留规则：不确定内容不强行改，进入待复核清单。

### 5.3 不确定项策略

不要强行“润色到看似通顺”。对以下内容应保留疑点：

- 无法确认的人名、公司名、矿山名、地名。
- 影响报价或合同的数字。
- 单位可能错识的数字。
- 是否为正式承诺的表达。
- 采矿资质、挂靠、合同、税务、套保等高风险内容。

## 6. 知识库与自学习

### 6.0 知识库位置策略

知识库分两层：

- 内置知识库：放在 `references/`，随 skill 发布和升级，包含通用矿业术语、默认错词规则、纪要模板和 prompt 模板。
- 私有知识库：放在 `knowledge/`，记录用户使用中积累的错词、人名、客户名、矿山名、候选学习结果和任务历史。

MVP 默认将私有知识库存放在项目目录内的 `knowledge/`。这样最简单、可复制、方便在不同设备或 agent 间整体迁移。

为了后续多设备共用，CLI 应预留 `--knowledge-dir` 参数。例如未来可以指定：

```bash
python scripts/transcribe_pipeline.py --input 转写源 --knowledge-dir /path/to/shared/knowledge
```

不建议第一版就做数据库或云同步，以免过早增加部署复杂度。

### 6.1 正式错词表

建议文件：`knowledge/confirmed_corrections.jsonl`

示例：

```json
{"wrong":"李辉时","correct":"锂辉石","domain":"矿物","confidence":1.0,"status":"confirmed","note":"矿业语境固定错词"}
{"wrong":"假常识","correct":"钾长石","domain":"矿物","confidence":0.98,"status":"confirmed","note":"选矿、尾矿、砂石上下文"}
```

### 6.2 候选学习表

建议文件：`knowledge/learned_candidates.jsonl`

示例：

```json
{"wrong":"淘宝","correct":"套保","confidence":0.91,"reason":"上下文出现期货、做空、手数、现货","source":"05-25 客户会议","status":"candidate"}
```

### 6.3 晋升规则

候选进入正式表的推荐条件：

- 用户明确确认。
- 在多次会议中重复出现。
- 上下文高度稳定。
- 目标词已经存在于术语库。
- 模型置信度高于阈值，例如 `0.95`。
- 不涉及人名、公司名、合同金额等高风险字段。

### 6.4 禁止自动晋升的内容

以下内容默认不能自动写入正式错词表：

- 人名。
- 公司名。
- 地名。
- 矿山名。
- 合同金额。
- 付款条款。
- 法律、合规、税务表述。

这些内容可以写入候选或待复核清单。

## 7. 大模型接入建议

### 7.1 DeepSeek 的角色

DeepSeek 或其他大模型适合做：

- 上下文纠错。
- 分段摘要。
- 会议纪要生成。
- 新错词候选提取。
- 待复核疑点归纳。

不适合单独负责：

- 无审计地修改正式知识库。
- 无校验地改动关键数字。
- 无来源地创造人名、公司名、承诺事项。

### 7.2 Prompt 基本结构

每个纠错片段应包含：

- 任务说明。
- 当前会议主题。
- 上下文摘要。
- 相关术语表片段。
- 已知错词映射。
- 原始转写片段。
- 输出格式要求。
- 不确定项处理规则。

输出建议为 JSON：

```json
{
  "corrected_text": "...",
  "changes": [
    {"from":"品味","to":"品位","reason":"矿业术语"}
  ],
  "uncertain_items": [
    {"text":"某某公司","reason":"可能为公司名，无法确认"}
  ],
  "learn_candidates": [
    {"wrong":"淘宝","correct":"套保","confidence":0.91}
  ]
}
```

## 10. MVP 开发路线

### 阶段 1：最小可用版本

- 创建通用 skill 目录结构。
- 实现 PDF/TXT 文本提取。
- 实现基础清洗。
- 建立初版术语表和错词表。
- 实现本地规则纠错。
- 输出基础会议纪要。
- 输出进度 JSON Lines。

验收标准：

- 能处理当前四份 PDF。
- 能输出纠错文本和会议纪要。
- 能列出关键数据、行动项和待复核项。
- 不依赖 macOS 专用命令。

### 阶段 2：DeepSeek 增强

- 接入 DeepSeek API。
- 支持 `local/agent/deepseek/auto` 模式。
- 实现分块纠错 prompt。
- 实现 JSON 输出解析和本地校验。
- 将新错词写入候选表。

验收标准：

- 有 API key 时能自动调用 DeepSeek。
- 无 API key 时能退回本地或 agent 模式。
- 不把候选错词直接写入正式表。

### 阶段 3：飞书 + OpenClaw 高频场景

- 提供明确 CLI 调用方式，必要时再封装 HTTP 服务。
- 输出适合飞书消息展示的进度。
- 支持 `/skill` 形式触发。
- 输出 Markdown 纪要、纠错全文和结构化附件。
- 飞书文档写入由后续部署飞书时单独设计，通过飞书 CLI 将 `04_minutes.md` 等结果写入飞书文档并保存到飞书云盘指定位置。

验收标准：

- OpenClaw 可通过 `/skill` 调用核心 CLI。
- 飞书侧能看到进度。
- 输出结果结构稳定，便于后续飞书 CLI 接入。

### 阶段 4：知识库维护

- 提供候选复核命令。
- 支持将候选晋升到正式错词表。
- 支持导出术语表。
- 支持按客户、矿山、会议类型维护扩展词库。
