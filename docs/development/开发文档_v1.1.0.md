## 2. 当前默认行为

根据用户最新判断，默认模式已调整为 DeepSeek 模式，因为样本验证显示大模型生成的会议纪要明显优于本地规则摘要。

当前 CLI 默认值：

- `--mode deepseek`
- `--deepseek-tier flash`
- `--progress text`
- `--template business`
- `--learn-policy candidate`
- `--max-chars 4500`

默认生产命令：

```bash
python3 sichuan-mining-transcribe/scripts/transcribe_pipeline.py \
  --input 转写源 \
  --output sichuan-mining-transcribe/outputs/run-001 \
  --mode deepseek \
  --deepseek-tier flash \
  --progress text \
  --knowledge-dir sichuan-mining-transcribe/knowledge \
  --learn-policy candidate
```


## 3. DeepSeek 接入设计

DeepSeek 配置已参考 2026-05-28 可访问的官方文档：

- https://api-docs.deepseek.com/zh-cn/
- https://api-docs.deepseek.com/zh-cn/guides/json_mode

当前实现位置：

- `sichuan-mining-transcribe/scripts/llm_correct.py`
- `sichuan-mining-transcribe/scripts/transcribe_pipeline.py`
- `sichuan-mining-transcribe/references/prompt_templates.md`

当前 API 约定：

- 默认 Base URL：`https://api.deepseek.com`
- Chat Completions Endpoint：`/chat/completions`
- JSON 模式：`response_format: {"type": "json_object"}`
- API Key 来源：环境变量 `DEEPSEEK_API_KEY`
- 可选 Base URL 覆盖：环境变量 `DEEPSEEK_BASE_URL`

模型分层：

| 等级 | 参数 | 模型 | thinking | 用途 |
|---|---|---|---|---|
| Flash | `--deepseek-tier flash` | `deepseek-v4-flash` | 默认 disabled | 日常默认，速度优先 |
| Pro | `--deepseek-tier pro` | `deepseek-v4-pro` | 默认 enabled | 高要求转写、复杂会议、正式归档 |

可用 `--model` 覆盖模型名，可用 `--deepseek-thinking enabled/disabled` 覆盖 thinking 行为。

安全注意：

- API key 不应写入仓库、文档或命令示例。
- 本轮测试中用户曾在对话中提供 key，后续建议轮换该 key。
- 后续部署时建议通过环境变量、服务器密钥管理或飞书/OpenClaw 的安全配置注入。

## 4. 会议纪要生成策略

用户反馈本地模式输出存在“按句子抽取、过散过杂”的问题，因此当前 DeepSeek 模式改为：

1. 先进行本地规则纠错。
2. 分块调用 DeepSeek 进行上下文纠错。
3. 将纠错全文交给 DeepSeek 做全文语义理解。
4. 用语义 JSON 结果生成 `04_minutes.md`。
5. 同步生成结构化 CSV 和疑点清单。

`references/prompt_templates.md` 已明确要求：

- 会议纪要必须按会议级别聚合，而不是逐句堆砌。
- 关键数据与商务条件只保留对决策有用的信息。
- 技术与生产要点按主题归纳。
- 风险与待确认事项按风险类型归纳。
- 行动项只保留明确需要后续执行的事项。
- 关键数字默认 `needs_review` 为“是”。

纠错 prompt 已补充限制：

- 纠错阶段只修正识别错误。
- 不扩写、不总结、不补充背景、不改写成纪要。
- `corrected_text` 长度应与原片段大体相当。
- 保留说话人、时间戳、段落顺序和口语信息。

这个限制是因为已测试的 flash 样本曾出现纠错全文从约 1.4 万字扩写到约 2.3 万字的问题。

## 5. 进度输出设计

用户要求长等待过程中实时输出进度，减少等待焦虑。当前已实现：

- 默认 `--progress text`：终端输出中文可读进度。
- `--progress jsonl`：终端输出机器可读 JSON Lines。
- `--progress none`：终端不输出，但仍写入 `run_log.jsonl`。
- 每次运行都会生成 `run_log.jsonl`。

文本进度示例：

```text
[总耗时 73.6s | 本阶段 73.5s] DeepSeek：DeepSeek 纠错进行中：第 2/5 段，已完成 6501/13958 字（46.6%）
```

JSONL 事件字段：

- `stage`
- `message`
- `done_chars`
- `total_chars`
- `elapsed_seconds`
- `stage_elapsed_seconds`
- DeepSeek 阶段额外包含 `model`、`thinking`、`chunk_index`、`chunk_total`

该设计便于：

- macOS/Windows 终端直接看。
- Codex/Claude Code 展示。
- OpenClaw 或飞书机器人转发进度。
- 后续 HTTP 服务或队列任务复用日志。

## 6. 输出文件协议

每次运行输出目录包含：

```text
01_extracted.txt
02_normalized.txt
03_corrected.md
04_minutes.md
05_key_data.csv
06_action_items.csv
07_uncertain_items.md
08_learn_candidates.jsonl
09_semantic_minutes.json    # DeepSeek 模式才有
manifest.json
rule_changes.jsonl
run_log.jsonl
```

关键文件：

- `03_corrected.md`：纠错全文。
- `04_minutes.md`：会议纪要，当前最重要产物。
- `09_semantic_minutes.json`：DeepSeek 语义纪要中间结果，便于后续调试模板。
- `08_learn_candidates.jsonl`：候选学习结果，不自动晋升正式错词表。
- `manifest.json`：本次运行的输入、模式、模型、输出文件和统计信息。

注意：早期样本目录中的 `manifest.json` 使用旧 schema，可能没有 `deepseek_used` 和 `deepseek_thinking` 字段。当前代码生成的新 manifest 已包含这些字段。

建议后续第一步：

```bash
python3 sichuan-mining-transcribe/scripts/transcribe_pipeline.py \
  --input "转写源/05-24 客户会议_ 锂矿合作洽谈.pdf" \
  --output sichuan-mining-transcribe/outputs/sample-deepseek-flash-0524-v2 \
  --mode deepseek \
  --deepseek-tier flash \
  --progress text \
  --knowledge-dir sichuan-mining-transcribe/knowledge \
  --learn-policy candidate
```

重点检查：

- `03_corrected.md` 是否仍异常扩写。
- `04_minutes.md` 是否保持语义聚合质量。
- `run_log.jsonl` 是否记录 `thinking: disabled`。
- `manifest.json` 是否记录 `deepseek_used: true`。

### 8.2 Pro 模式需要按新参数复测

当前代码已支持：

- `deepseek-v4-pro`
- thinking 默认 enabled
- `reasoning_effort: high`

但尚未按新代码复跑 pro。建议用 05-24 样本复测，再视成本决定是否跑四份全量样本。

### 8.3 本地模式仍可保留但不作为质量默认

本地模式价值：

- 无网络。
- 隐私最高。
- 成本最低。
- DeepSeek 不可用时兜底。

但正式纪要质量明显弱于 DeepSeek。后续可以继续优化本地模式，但优先级低于 DeepSeek prompt 和稳定性。

### 8.4 Feishu/OpenClaw 集成暂未实现

当前仍只生成本地文件。后续飞书部署时再做：

- 调用飞书 CLI 创建/更新飞书文档。
- 保存到飞书云盘指定目录。
- 将 `04_minutes.md` 写入正文。
- 将 CSV 或 JSON 作为附件或表格。
- 把 `run_log.jsonl` 的进度转成飞书消息。

### 8.5 知识库晋升机制仍是 MVP

当前已输出候选学习，但还没有完整的审核/晋升工具。后续可加：

- `review_candidates.py`
- `promote_candidates.py`
- `knowledge/confirmed_corrections.jsonl`
- 重复候选聚合统计。

正式映射表仍不能让模型自动改写，避免污染。
