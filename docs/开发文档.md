# 四川矿业会议转写 Skill 开发文档

最后更新:2026-06-12

本文档说明项目背景与当前(重构后)的开发状态,供继续开发或交接时快速恢复上下文。
原始需求见 [任务背景.md](任务背景.md),面向使用者的说明见根目录 [README.md](../README.md)。

## 1. 项目背景

用户是四川乐山的矿业企业负责人,会议参与方多说四川话、乐山话,夹杂普通话。通用语音转写工具
对乐山话和矿业术语的识别错误率很高。本 skill 的职责是:**接收已由其他工具转写好的文本
(PDF/TXT/MD),做四川话/乐山话/矿业术语纠错,并生成结构化会议纪要。**

核心诉求(摘自任务背景):

- 全流程连续,从提供转写源到产出纪要,中途无需人工干预。
- 长任务持续反馈进度(超过 1 分钟要让用户看到"还在跑")。
- 兼容 PDF 与 TXT 输入。
- 重点优化矿业领域(锂矿、选矿、品位、回收率、精粉、套保、报价、付款、资质等)用词。

## 2. 当前架构(2026-06 重构后)

历史版本是 7 个脚本组成的流水线(extract / normalize / apply_rules / llm_correct /
generate_minutes / update_knowledge / transcribe_pipeline),输出 11 个文件,存在跑不完、
无进度反馈、候选知识只进不出等问题。**已整体重构为单文件引擎。**

整个 skill 自包含在 `sichuan-mining-transcribe/` 目录,拷走即用:

```text
sichuan-mining-transcribe/
├── SKILL.md                 # skill 入口与部署自检清单
├── transcribe.py            # 唯一核心引擎(提取→清洗→纠错→纪要→自进化)
├── requirements.txt         # PyYAML、pypdf
├── .env / .env.example      # DEEPSEEK_API_KEY(.env 本地保留,不进库)
├── knowledge/               # 纠错知识库,随 skill 一起部署
│   ├── glossary.yml         # 矿业术语表,注入纠错提示词
│   ├── confirmed_rules.yml  # replacements 硬替换 + context_hints 上下文提示
│   └── candidates.jsonl     # 自动累积的候选纠错对(带频次)
└── outputs/runs/<日期-会议名>/   # 每次运行产出,默认不进库
    ├── 纠错全文.md
    ├── 会议纪要.md
    └── 疑点清单.md
```

### 处理流程

1. **提取**:PDF 用 pypdf,TXT/MD 多编码兜底解码;支持单文件或目录。
2. **清洗**:NFKC 归一、空白与换行规整。
3. **本地规则纠错**:`confirmed_rules.yml` 的 `replacements` 做零成本硬替换。
4. **会议背景概览**:纠错前先跑一次轻量请求(`generate_overview`),从开头约 4000 字
   提炼会议主题、矿种、参会方,作为全局上下文注入每一段纠错提示词 —— 弥补并行分段
   各自独立、看不到前后文的缺陷,让方言术语判断有据可依。
5. **DeepSeek 并行纠错**:分段(默认 3500 字)后 6 路并发,每段逐一落盘到 `.checkpoint/`。
   - 单段超时 90 秒、重试 2 次仍失败则回退本地版,并记入疑点清单 —— 流程必定跑完。
   - 返回长度偏离原文 0.6~1.4 倍判为异常,同样回退,防止扩写/空文。
   - 中断后用同一输出目录重跑可断点续传,已完成段不重算。
6. **纪要生成**:长文分节并行摘要 → 汇总成稿,避免单请求塞入全文。
7. **进度心跳**:后台线程每 5 秒打印阶段、已完成字数/段数、百分比和预计剩余时间。

### 会议纪要模板与提示词

提示词是质量核心,全部以常量形式定义在 `transcribe.py`,无独立模板文件:

- `OVERVIEW_PROMPT` —— 会议背景概览。
- `CORRECTION_PROMPT` / `CORRECTION_SYSTEM` —— 逐段纠错;含安全红线(不改数字、人名等)。
- `SUMMARY_PROMPT` —— 分段提炼要点;要求数字逐字保留并附原文短句。
- `MINUTES_PROMPT` —— 汇总成稿,纪要结构固定为:摘要 / 会议信息 / 核心结论与共识 /
  关键数据(表格:类别 \| 数值 \| 原文摘录 \| 说明)/ 商务条件 / 行动项 / 风险与分歧 /
  待人工复核疑点。

纪要质量上的几个关键设计:

- **会议信息**的日期、议题由 `parse_meeting_name` 从文件名解析(如 `06-12 矿山合作` →
  日期 `06-12`、议题 `矿山合作`),不再常年"待补充"。
- **关键数据可溯源**:每个数字带"原文摘录"列,核对时无需回翻全文。
- **疑点精准**:纠错阶段产生的存疑项(回退段、数字变更等)经 `format_uncertain_digest`
  传给纪要提示词,只对这些及真正有歧义处标"(待核实)",避免旧版"满屏待核实"。
- **发言人解析**:要求用对话中的称呼(如"曹总"),不把"发言人N"当参会方列出。
- 改模板:直接编辑上述常量即可;若改"## 待人工复核疑点"标题,需同步
  `extract_minutes_uncertain` 的正则,否则疑点清单会抽空。

### 关键参数

| 参数 | 默认 | 说明 |
|---|---|---|
| `--tier` | `flash` | `flash` 日常;`pro` 重要会议,质量更高约慢一倍 |
| `--local` | 关 | 离线模式,只跑本地规则,不调 API |
| `--workers` | 6 | 并行请求数 |
| `--max-chars` | 3500 | 纠错分段长度 |
| `-o` | 自动 | 输出目录,默认 `outputs/runs/<日期-会议名>` |

## 3. 自进化机制

目标是"越用越准",且产生与生效尽量自动,只有最后晋升为硬规则需人工点头:

1. **自动提取**:每次运行 diff 比对本地规则版与模型纠错版,提取短替换对(1~8 个汉字)。
   涉及数字/金额的变更不进候选,改入疑点清单。
2. **频次累积**:写入 `candidates.jsonl`,记录出现次数与涉及的会议(按会议名去重)。
3. **自动生效(软)**:在 ≥2 场会议中出现的候选,自动注入后续运行的纠错提示词。
4. **一键晋升(硬)**:`python transcribe.py review` 逐条 y/n 审核;采纳的多字词进
   `replacements` 成为硬替换,单字词只进 `context_hints`(避免无上下文的全局误替换)。

知识库随 skill 目录一起部署,纠错经验在用户各处的部署间共享。

## 4. 跨平台部署

- **Claude Code**:`cp -r sichuan-mining-transcribe ~/.claude/skills/`(或放项目 `.claude/skills/`)。
- **Codex / OpenClaw / Hermes 等**:能力本质是单文件 CLI,平台即使不识别 SKILL.md 格式,
  让 agent"读取 SKILL.md 并照其执行"即可。SKILL.md 顶部含首次运行自检清单
  (Python ≥3.9、装依赖、`.env` 有 key、网络可达 api.deepseek.com)。

## 5. 隐私约定

- 不提交:`.env`、真实会议转写源、`outputs/`。
- 跟随 skill 提交:代码、术语规则、`knowledge/` 全部知识(本项目仅个人使用,知识需随包走)。
- 安全红线:数字、金额、日期、人名、公司名、矿山名不做静默改动,可疑处一律进疑点清单。

## 6. 后续可做

- 飞书 / OpenClaw 集成:`--progress` 已可机器解析,可把进度回传群消息,纪要回写文档。
- 候选累积去重:同一会议重跑会重复累加 `count`(不影响"≥2 场"生效门槛,仅影响 review 排序),
  如需精确可在写入前按"会议名+替换对"去重。
- review 工具可加批量采纳/导出,方便定期整理知识库。
